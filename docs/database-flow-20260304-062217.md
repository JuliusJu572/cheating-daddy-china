# 数据库流程说明（后端）

更新时间：2026-03-04 12:35:00

## 1. 目标与范围

本文说明 `user-management` 后端与 PostgreSQL 的数据流，重点包括：

- 认证与用户角色
- 简历/录音/会话等核心表
- 与客户端的交互边界
- 当前“本地优先”策略下数据库的职责

## 2. 认证与用户主链路

### 2.1 用户表（`users`）

关键字段：

- `id`
- `email`
- `password_hash`
- `license_key`
- `role`（`user` / `admin`）
- `created_at` / `updated_at`

约束：`email` 与 `license_key` 至少有一个非空（`CHECK (email IS NOT NULL OR license_key IS NOT NULL)`）。

### 2.2 认证接口（`/auth/*`）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/auth/register` | POST | 普通用户邮箱注册，密码至少 8 位，写入 `users(email, password_hash, role='user')` |
| `/auth/login` | POST | 普通用户邮箱登录，校验 `password_hash` 后签发 JWT |
| `/auth/license` | POST | License Key 登录，无则自动创建用户（仅 `license_key`） |
| `/auth/admin-register` | POST | 管理员注册，仅 `ADMIN_EMAILS` 中的邮箱可注册为 `role=admin` |
| `/auth/admin-login` | POST | 管理员登录，仅 `role=admin` 可登录管理后台 |
| `/auth/me` | GET | 校验 JWT，返回当前用户信息（需 `Authorization: Bearer <token>`） |

### 2.3 鉴权流程

1. 客户端调用上述 `/auth/*` 接口完成认证。
2. 服务端签发 JWT（有效期 7 天）。
3. 客户端后续带 `Authorization: Bearer <token>` 访问受保护接口。

## 3. 业务数据主表

## 3.1 简历表（`resumes`）

用于存储与简历相关的信息（历史/兼容/后台管理视角）。  
当前产品策略下，简历“主注入链路”在客户端本地完成，但后端仍保留相关接口和数据能力。

## 3.2 用户画像表（`user_profiles`）

用于用户基础资料扩展（展示名、头像、简介等）。

## 3.3 录音表（`voice_recordings`）

保存录音元数据（路径、时长、来源、创建时间等），用于记录管理与回溯。

## 3.4 面试会话表（`interview_sessions`）

保存面试会话级信息（语言、标题、状态、总轮次、起止时间）。

## 3.5 面试回答表（`interview_responses`）

保存每轮问答与关联信息（题目、回答、截图路径、轮次等）。

## 4. 与客户端的数据流关系

### 4.1 有账号态（已登录）

- 客户端带 token 可访问后端资源。
- 数据可落库并通过后台查看/管理。
- 客户端本地缓存按账号隔离。

### 4.2 无账号态（仅 License Key）

- 可使用本地 AI 能力（简历解析、JD 解析、ASR、回答）。
- 不依赖后端 token，不走账号数据同步链路。
- 简历/JD 按当前策略不持久化缓存（重启后需重传）。

## 5. 当前职责边界（给协作方）

- 客户端负责：实时交互、上下文拼装、ASR热词增强、未登录本地流程。
- 后端负责：认证鉴权、用户与记录管理、管理员视图、结构化数据留存。
- 两端通过 JWT + REST 接口协作；是否落库取决于是否处于登录账号态。

## 6. 运维与安全注意项

- 生产必须配置强 `JWT_SECRET` 与数据库访问控制。
- 管理员注册受 `ADMIN_EMAILS` 配置约束（若启用）。
- 禁止在日志中输出明文密码、完整 token、完整密钥。
